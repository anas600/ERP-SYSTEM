1. ليش اخترنا Squash Merge؟

Squash Merge يأخذ كل commits الـ feature branch ويضغطها في commit واحد على الفرع الهدف. هذا يحافظ على تاريخ نظيف على main (production) دون pollution من commits العمل الداخلي.

المقارنة المرئية بين 3 استراتيجيات

جدول مقارنة شامل

المعيار Squash ✅ Merge Commit Rebase
عدد commits على main1 لكل PRN+1N
تاريخ نظيف على main✅ نعم❌ pollution✅ خطي
rollback feature✅ revert 1 commit✅ revert merge⚠ revert N commits
changelog واضح✅ PR body = release note❌ commits متفرقة⚠ commits بدون merge
التاريخ التفصيلييبقى على develop✅ محفوظ على main❌ SHA يتغير
CI verification✅ PR يفحص integration✅ نفس✅ نفس
الـ Co-authors✅ في squash commit body✅ في كل commit✅ في كل commit

ليش اخترنا Squash لمشروعك؟

1. أنت solo developer

ما تحتاج commits فردية. تفضل تشوف "Phase X released" كحدث واحد في تاريخ main، بدل 12 commits داخلية.

2. main = production

كل commit على main = release candidate. Squash يخلي الـ release notes (commit body) تحوي كل التفاصيل.

3. develop = integration lab

التاريخ التفصيلي يبقى على develop. لو احتج تشوف commits الفردية: git log origin/develop.

4. Rollback سهل

لو Phase 4 فيه bug: git revert <squash-commit-hash> يرجّع كل التغييرات في خطوة واحدة.

4. كيف يفهم الـ Workers الـ Workflow؟

السؤال الجوهري: كيف يقرأ الـ worker التعليمات ويفهم السياق؟

📚 المصادر اللي يقرأها الـ worker قبل البدء

1. AGENTS.md (root)

يعرف الـ stack، الـ conventions، الـ branches strategy.

2. docs/PLAN.md

يعرف الـ phase الحالية وما هو مطلوب.

3. docs/research/gap-analysis.md

يعرف scope الـ feature — كم entity، كم endpoint، كم migration.

4. الـ AGENTS.md الخاصة بالـ module

يعرف patterns الموجودة (entity layout، repository pattern، DI).

📨 الـ Prompt اللي يستقبله الـ worker

كل worker task في الـ plan يحوي prompt واضح بـ:

// الـ prompt structure (من plan_b5ae4fc0)
prompt: |
  أنت كبير مهندسين Backend (.NET 9 + Dapper + PostgreSQL). اقرأ:
  - docs/research/gap-analysis.md (خاصة قسم Phase 4)
  - src/backend/AGENTS.md
  - src/backend/Modules/Inventory/AGENTS.md (نفس النمط)

  المطلوب: بدون كسر الـ existing.

  ## A) Module Name
  1. Migration جديد: src/backend/Modules/X/Infrastructure/Migrations/...
  2. Entities: ...
  3. DTOs + Validators: ...
  4. Services: ...
  5. Controller: ...

  ## Constraints
  - لا تغيّر أي ملف في Identity, Finance, Projects, ...
  - dotnet build → 0 errors

  ## Output
  - Branch: feature/phase-4-X
  - Conventional commits (5-8 commits)
  - PR إلى develop

  Stop condition: PR مفتوح + Backend يعمل + smoke test ناجح.

🔄 كيف الـ worker يعرف أن الـ branch نظيف؟

  1. الـ worker يبدأ بـ:
    git checkout develop && git pull origin develop && git checkout -b feature/X

    هذا يضمن أن الـ branch يبدأ من آخر integration.

  2. بعد كل commit:
    git push origin feature/X

    الـ PR يُفتح تلقائياً (سواء من worker أو من Mavis).

  3. بعد merge في develop:

    branch cleanup rule يقول للـ worker (ولـ Mavis): احذف الـ branch فوراً.

  4. قبل بداية feature جديدة:

    الـ worker (أو Mavis) يعمل git fetch + git pull للتأكد من وجود آخر commits على develop.

🧠 كيف الـ worker يفهم "clean history"؟

الـ worker لا يحتاج أن يفهم — يكفي التعليمات الصريحة في الـ prompt:

// في الـ prompt نحدد:
Branch: feature/phase-4-payroll-engine
Commits: Conventional commits (5-8 commits)
PR: افتح PR إلى develop
بعد merge: المالك (Mavis) يحذف الـ branch تلقائياً

// الـ worker يفهم:
- هو يكتب على feature/X
- هو يصنع PR إلى develop
- لا يفتح PR إلى main أبداً
- لا يحذف branches (هذا من مسؤولية Mavis)

// المالك (Mavis) يتأكد:
- بعد كل merge: حذف branch + remote prune
- تحديث AGENTS.md (نفس قاعدة Phase 3)
- Push develop → main (Squash) في release النهائي

🤝 العقد بين المالك والـ Workers

⚠ العقد: الـ worker لا يدمج إلى main أبداً، ولا يحذف branches. المالك (Mavis) هو المسؤول الوحيد عن:
  • Merge feature → develop
  • Delete feature branches (local + remote)
  • PR develop → main (Squash فقط)
  • تحديث AGENTS.md + CHANGELOG + PLAN.md
✅ ما يحتاجه الـ worker فقط:
  • Branch feature (من develop)
  • Commits + Conventional prefixes
  • Push + PR إلى develop
  • Stop condition واضح (build clean + smoke test PASS)

- اضغط "Merge" → اختار "Squash and merge"

# أو من CLI
